# Q&A 02 — 需求讨论

**时间**：2026-03-22
**阶段**：Phase 1 需求讨论

---

## 问题1：输出形态

**问**：输出纯 HTML string / React 组件 / Virtual DOM / 抽象 AST？

**答**：渲染层输出 **AST 抽象**，渲染部分根据不同平台单独做，把解析和渲染完全解耦。

**含义**：
- 解析器只负责产出平台无关的 AST
- 不同平台（Web DOM、React、Vue、终端、Native 等）各自实现渲染适配器
- 解析器与渲染器之间通过 AST 接口约定通信

---

## 问题2：目标环境

**问**：浏览器端 / SSR+Node.js / 两者都要？

**答**：没有理解这个问题的含义，需要解释。

**解释**：
- **浏览器**：可操作 DOM，有 `window`/`document`，bundle size 敏感
- **Node.js/服务端**：无 DOM，常用于 SSR（服务器渲染成 HTML 字符串）
- **两者都要**：不能依赖 DOM API，或需要分别适配

由于渲染层按平台单独做，解析层本身要求**平台无关**，这与语言选型直接挂钩。

---

## 问题3：Markdown 规范范围

**问**：GFM / CommonMark 子集 / AI 场景常见子集？

**答**：要求**大而全**，都要有能力接入。解析处通过**组件拆分动态组合**，决定对外提供能力时的解析及渲染的能力边界，要求**能力可插拔**。

**含义**：
- 以 **GFM（GitHub Flavored Markdown）** 为基准
- 额外支持**自定义解析器**和**自定义渲染器**的注册/组合
- 各个语法特性作为独立插件存在，可按需组合

## 问题5：性能目标

**问**：每个 token 到来的处理延迟要求？是否有大文档场景？

**答**：先按 **50ms 一次新 token** 来考虑，同时需要考虑大文档场景。

**分析结论**：50ms 过于保守，详见 [03_technical_decisions.md](./03_technical_decisions.md)

## 已确认的核心需求

| 维度 | 决策 |
|---|---|
| 输出 | AST 抽象，平台无关 |
| 解析/渲染关系 | 完全解耦 |
| Markdown 基准 | GFM |
| 扩展性 | 能力可插拔，支持自定义解析器/渲染器 |
| 性能 | 按 <5ms/token 设计（见技术讨论） |
| 大文档 | 需要支持 |
