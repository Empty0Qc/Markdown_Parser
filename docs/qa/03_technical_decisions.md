# Q&A 03 — 技术选型讨论

**时间**：2026-03-22
**阶段**：Phase 1 → Phase 2 过渡

---

## 目标环境结论

解析器要求**平台无关**（不依赖任何运行时环境的特定 API），原因：
- 渲染层按平台单独实现
- 解析器本身只做文本处理，无需 DOM 或 OS 接口

---

## 语言选型对比

| | **TypeScript** | **Rust + WASM** | **C/C++ + WASM** |
|---|---|---|---|
| 性能 | 中 | 极高 | 极高 |
| 流式/增量友好 | 好（micromark 原生支持） | 好（手写状态机） | 中（cmark 非流式设计） |
| WASM 跨界成本 | 无 | 有（字符串序列化） | 有 |
| 插件/可扩展性 | 极好 | 中（需要暴露 FFI） | 差 |
| AST 跨边界传输 | 天然 JS 对象 | 需要设计协议 | 需要设计协议 |
| 调试难度 | 低 | 中 | 高 |
| 构建复杂度 | 低 | 中 | 高 |

### Rust+WASM 的根本矛盾

流式增量意味着**高频跨 WASM 边界调用**，每次 token 到来都要 JS → WASM → JS。字符串编解码的开销会吃掉性能优势。除非把状态完全驻留在 WASM 内存里只传 delta，但这大幅增加架构复杂度。

### 各方案推荐分析

- **TypeScript**：零序列化成本，`micromark` 原生流式，插件系统自然，性能够用
- **Rust+WASM**：极致性能，但跨边界开销和架构复杂度高
- **C**：极致性能，平台无关，可编译为 WASM/native library，内存管理需要严格设计

---

## 决策结果

### Q1：解析器语言 → **C**

**理由**（用户选择）：极致性能，平台无关，可通过 Emscripten 编译为 WASM 供浏览器使用，也可作为 native library 供 Node.js/其他平台调用。

**需要考虑的约束**：
- 插件/可扩展性需要通过函数指针/回调机制实现
- AST 跨边界传输需要设计序列化协议（C → host 语言）
- 内存管理需要明确所有权策略

### Q2：Markdown 规范 → **GFM + 自定义扩展**

- 以 GFM（GitHub Flavored Markdown）为基准
- 支持自定义解析器注册
- 支持自定义渲染器注册
- 各语法特性作为独立模块，可按需组合

## 性能目标讨论

### 用户初始假设：50ms/token

### 实际 LLM 速度

| 模型 | 典型速度 | 对应 token 间隔 |
|---|---|---|
| GPT-4o | 50–100 tok/s | 10–20ms |
| Claude Sonnet | 80–120 tok/s | 8–12ms |
| 本地模型（llama.cpp GPU） | 100–200 tok/s | 5–10ms |
| 未来趋势 | 只会更快 | — |

### 结论：50ms 过于保守

**解析器设计目标：`<5ms/token`**（覆盖当前及近未来所有主流模型速度）

### 关键架构洞察：解析与渲染频率解耦

```
token stream (5–20ms/tok)
        ↓
    Parser（每个 token 都跑，维护状态）
        ↓
  AST delta queue（解耦缓冲层）
        ↓
  Renderer（按帧预算跑，16ms/frame for 60fps）
```

- 解析器：不受渲染帧率限制，逐 token 处理
- 渲染器：按浏览器帧率（16ms）批量消费 AST delta，避免抖动
- 两者通过 **delta queue** 解耦
